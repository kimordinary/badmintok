4. 실서버(Django API) 풀세션 E2E

✅ 통과. 실제 백엔드 API로 20명/4코트 세션을 시작→출석→코트채움→경기종료 30라운드 자동반복까지 돌려, 카운트 균등(편차 ≤3)·매칭 유효(4명 중복 없음)·다양성을 검증. 백엔드 전체 46개 테스트 OK. (시뮬은 core.jsx, 이건 진짜 서버 경로)

종합 결론

  • 알고리즘은 출시해도 될 만큼 견고 — 급수분포·성비·벤치 어떤 조합(150)에서도 균등·박빙 유지. 실서버 경로도 E2E 통과.
  • 한계 1 — 벤치 0~1: 대기 인원이 없으면 다양성·박빙 붕괴. 운영 가이드로 "벤치 ≥2~4명" 권장하면 회피.
  • 한계 2 — 혼복만 + 여 25% 미만: 성비 쏠리면 혼복 불가. 현실 제약이라 "모두/남복여복"으로 우회(프로그램은 알림으로 대응).
  • 이 둘 다 알고리즘 버그가 아니라 "물리적 한계" — 인원/성비가 근본적으로 부족한 경우. 운영 가이드로 커버 가능.
  • 남은 출시 리스크는 알고리즘 밖 — 실기기 UI·실제 사람 행동·파일럿 운영. 이건 자동 검증 불가, 실제 1회 운영 필요.
🚀 결론: 알고리즘·백엔드 = 출시 가닥 잡힘. 운영 가이드(벤치 ≥2, 성비 따라 종목 선택)만 문서화하고, 실제 파일럿 1회면 제품 출시 판단 가능.
